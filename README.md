# 수집형 RPG 가챠(Gacha) 서버 API - 성능 및 정합성 최적화

트랜잭션 정합성을 보장하며, 대규모 동시 접속(신규 픽업 배너 오픈) 시 발생하는 RDBMS 병목을 분석하고 최적화한 가챠 서버 API 프로젝트입니다.

## 🛠 Tech Stack
* **Backend:** C# .NET 10, Entity Framework Core
* **Database & Cache:** PostgreSQL, Redis
* **Test & Metrics:** k6 (Load Testing), Postman, swagger
* **Security:** JWT (JSON Web Token)

---

## 🚀 핵심 아키텍처 및 비즈니스 로직

### 1. 서버 권위(Server-Authoritative) 아키텍처 구현
* **클라이언트 변조 원천 차단:** 게임 클라이언트를 신뢰하지 않고, 비용(`Cost`)과 유저 식별자(`UserId`)의 주도권을 서버로 이관했습니다.
* **JWT 기반 무상태성 인증:** HTTP 헤더의 서명된 토큰에서 직접 ID를 추출하여 타인 명의 도용(Spoofing)을 방어했습니다.
* **서버 주도 결제:** 클라이언트가 전송한 가격이 아닌, 서버 DB에 적재된 배너 마스터의 정가를 대조하여 재화를 차감합니다.

### 2. ACID 트랜잭션 및 원장 정합성 보장
* 재화 차감(UPDATE), 인벤토리 아이템 지급(INSERT), 가챠 로그 기록(INSERT)의 3단계 디스크 쓰기 작업을 단일 트랜잭션으로 묶어, 동시 다발적인 요청 속에서도 재화 복사 버그나 데드락 없이 100%의 무결성을 보장합니다.

### 3. Redis Cache-Aside 패턴을 통한 Read 병목 제거
* 가챠 확률 메타데이터의 '높은 읽기 빈도'와 '정적 특성'을 활용하여 Redis 인메모리 캐싱을 적용했습니다. 
* 식별자 기반(BannerId)으로 캐시 키를 격리하고, DB 조회 부하(Select Query)를 0에 가깝게 최적화했습니다.

## 📊 부하 테스트 및 성능 최적화 (Troubleshooting)
오픈소스 부하 테스트 도구 **k6**를 활용하여 가상 유저(VU) 50명이 30초간 쉴 새 없이 가챠를 요청하는 스트레스 테스트를 3단계에 걸쳐 진행 및 개선했습니다.

### Phase 1. 논리적 락(Lock) 경합 분석
* **현상:** TPS 16.7 / 최대 응답 지연 19.36초
* **원인:** 단일 테스트 계정에 요청이 집중되어, PostgreSQL이 데이터 무결성을 위해 해당 유저 행(Row)에 배타적 잠금(Exclusive Lock)을 걸어 순차 처리 발생.
* **조치:** 각 가상 유저가 고유한 ID와 토큰을 발급받아 분산 접속하도록 스크립트 수정.

### Phase 2. 다중 접속 분산 및 물리적 I/O 한계 도달
* **현상:** TPS 22.7 / 최대 응답 지연 8.28초 (성능 2배 향상)
* **원인:** 논리적 락 경합은 해소되었으나, 1회 가챠당 3번의 무거운 디스크 I/O가 발생하여 DB 커넥션 풀 및 스레드 블로킹(Blocking) 발생. 특히 EF Core의 동기적 콘솔 로깅이 심각한 병목 유발.

### Phase 3. 스레드 블로킹 해소 및 최종 최적화
* **조치:** `appsettings.json`에서 EF Core의 SQL 콘솔 동기 출력(Logging)을 차단하여 I/O 병목 제거.
* **최종 결과:**
  * **초당 처리량(TPS):** 16.7 -> **402.6 (약 2,400% 향상)**
  * **평균 응답 지연:** 2,830ms -> **23.61ms (약 120배 개선)**
  * **에러율(Failed):** 0% (완벽한 트랜잭션 처리)

## 💡 향후 개선 목표 (Future Works)
* 현재 RDBMS 직접 쓰기(Direct Write) 방식의 한계를 극복하기 위해, **비동기 메시지 큐(Kafka, RabbitMQ)**를 도입하여 Write 부하를 워커 서버로 분산(Batch Insert)하는 아키텍처로 고도화할 계획입니다.
